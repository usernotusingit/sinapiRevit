O problema

Dois mundos sem chave em comum

O modelo Revit descreve o que existe (paredes, pisos, esquadrias). O SINAPI descreve quanto custa. Mas não há chave natural entre eles.

  • No Revit, o nome do tipo é texto livre do projetista — “Parede básica - int - ARRIMO (---- / TINTA)”. Carrega o acabamento, não a especificação do bloco nem um código.
  • No SINAPI, cada item é um código de catálogo do governo — disjunto do Revit. Sem chave estrangeira, sem campo compartilhado para um JOIN direto.
A pergunta central não é “qual fórmula?”, e sim “como construir a relação lógica entre os dois relatórios de forma confiável e auditável?”.

A solução

Uma ponte dimensional congelada

Como não existe chave natural, ela é fabricada: uma dimensão-ponte (crosswalk) construída uma vez, revisada e congelada. O custo passa a ser um JOIN puro.

dim_revit_typetipo + material + medida
→
crosswalkponte revisável
(a chave manufaturada)
→
dim_sinapi_composicaocódigo + custo por UF/regime
Toda decisão de correspondência fica na ponte. O motor de custo (fato) é apenas quantidade × custo unitário — um JOIN determinístico em DuckDB.
Modelo estrela (star schema)map_revit_to_sinapi → fact_orcamento

Correspondência

Determinismo por construção

  • Trava de unidade — um tipo medido em m²/m/un só pode casar com composições da mesma unidade. Impede erros grosseiros (ex.: janela em m² nunca casa com item em unidade).
  • Âncora de grupo — cada grupo Revit é restrito ao(s) grupo(s) SINAPI cabível(is). Forros só casam com “Forros/Gesso”; cobertura com “Telhamento”; e assim por diante.
  • Âncora de espessura (alvenaria) — a espessura da parede define a largura do bloco (9/14/19 cm). Necessária porque o nome da parede traz só o acabamento, não o bloco.
  • Ranqueamento fuzzy — similaridade textual dentro do conjunto já delimitado; empate pelo menor código. Espaço de busca pequeno e semântico ⇒ correspondência estável.
4 camadas + revisãounit gate → grupo → thickness → fuzzy
Uso de IA

Onde o LLM entrou — e onde não

O LLM foi usado uma vez, offline, só para resolver os ~41 tipos sinalizados pelo fuzzy: escolher o código SINAPI correto entre os candidatos pré-selecionados de cada tipo (ou marcar lacuna).

As decisões viram dados estáticos (apply_review.py + review_log.csv), com justificativa por linha. O JOIN de custo nunca chama LLM.

Premissas explícitas (proxies)

  • Vaso sanitário → bacia com caixa acopladapadrão para edifício público
  • Parede pintada → látex acrílica, 2 demãos
  • Piso cerâmico 40×40 → esmaltado 45×45dimensão mais próxima disponível
  • Guarda-corpo de vidro → painel panorâmico

Lacunas e limitações

12 lacunas — precificar manualmente

  • Calha de coberturaSINAPI 2026-05 não traz composição de calha
  • Portões metálicosexiste só por m² (106463); quantidade em metro
  • Divisórias de granito / naval, pele de vidro, laje aparente

Excluídas do total — sinalizadas, nunca “chutadas”.

Premissas estruturais do modelo

  • Alvenaria ancorada na espessura, não na especificação do bloco
  • Área de acabamento de parede = uma face calculadaverificar se devem ser as duas faces
  • Elementos estruturais não modelados neste projeto
A planilha herda os indicadores de confiança e quantity_status do modelo — números “moles” permanecem visíveis para revisão.
